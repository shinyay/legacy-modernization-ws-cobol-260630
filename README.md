# GitHub Copilot Quest Lv.3 — COBOL Modernization to Azure

> **Mission:** この中にある「ドキュメントもコメントも無い」レガシー COBOL アプリケーションを、
> **GitHub Copilot** と AI エージェントを駆使して読み解き、理解し、**Azure へモダナイズ**せよ。

---

## 🎯 このワークショップについて

**GitHub Copilot Quest Lv.3** は、GitHub Copilot を本気で使い倒して
**レガシー COBOL アプリケーションのモダナイゼーション** に挑戦するハンズオンイベントです。

ゴールはシンプルです — **この COBOL アプリケーションを Azure 上の姿に作り変えること。**

ただし、従来型の COBOL マイグレーションのセオリー（逐語的な
**Straight Conversion**、機械的な **Shift / Lift & Shift**、行単位の置き換え）に
**とらわれる必要はありません。** むしろ、

- AI エージェントに **コードを解析させて仕様を復元する**
- AI と対話しながら **ドメインを再発見し、あるべき姿を再設計する**
- **生成 AI ならではの自由な発想** で、新しいアーキテクチャに飛躍する

——そんな **AI ネイティブなモダナイゼーション** にチャレンジしてほしい、というのがこの Quest の狙いです。
「正解」は用意されていません。**あなたと Copilot が出す答えが正解です。**

---

## 🗺️ あなたのミッション

このリポジトリは、**意図的にドキュメントとコメントを取り除いた**「ナレッジが失われた現実のレガシーシステム」です。
仕様書はありません。コード内の説明コメントもありません。**動く実行コードだけ** があります。

現場で COBOL 資産を引き継ぐとき、まさにこの状態に直面します。だからこそ —

1. **解析する (Analyze)** — Copilot にコードを読ませ、何をしているのかを問いかける。
2. **理解する (Understand)** — 業務ドメイン・データの流れ・バッチの構造を、AI と一緒に再構築する。
3. **モダナイズする (Modernize)** — 理解をもとに、Azure 上のモダンな実装へと作り変える。

どこから手を付けるか、どんな Azure サービスに載せるか、どこまで作り変えるかは **あなた次第** です。

## 💡 進め方のヒント（Copilot を使い倒す）

決まったやり方はありませんが、行き詰まったらこんな使い方を試してみてください。

- **エージェントに探索させる**: 「このリポジトリは何をするアプリ？」「エントリポイントはどこ？」から始める。
- **仕様を復元させる**: 特定のプログラムを指して「このプログラムの入出力と業務ルールを説明して」。
- **可視化させる**: 「データフロー / バッチの依存関係を Mermaid 図にして」。
- **ドメインを再発見させる**: 「このスキーマとコードから推測されるビジネスドメインは？」
- **目標像を描かせる**: 「これを Azure に載せるなら、どんなアーキテクチャが考えられる？選択肢を比較して」。
- **作り変えさせる**: 言語移行・API 化・コンテナ化・サーバーレス化・データ移行 …… 発想は自由に。

> 🧭 まず **コードを動かして観察する** ところから始めると、AI への問いかけの解像度が上がります（下記参照）。

## 🛠️ まず動かしてみる（Build & Run）

このアプリは **Dev Container** で動くように構成されています。設計の説明はありません —
**ビルドして、走らせて、出力を観察する** ところから解析を始めてください。

### 前提環境

- **Visual Studio Code + Dev Containers 拡張**（または **GitHub Codespaces**）
  - `.devcontainer/` にツールチェーン一式（COBOL コンパイラ、PostgreSQL、メッセージブローカ等）が定義されています。
- Dev Container を使わない場合の主なランタイム: **GnuCOBOL**, **OCESQL**, **PostgreSQL**, **RabbitMQ**。

### 手順

```bash
# 1) リポジトリを Dev Container / Codespaces で開く（ツールチェーンが自動で揃います）

# 2) データベースのスキーマを用意する
make migrate

# 3) すべてのプログラムをビルドする
make build-all

# 4) ユニットテストを一括実行する（挙動を観察する格好の入口）
make test-all

# 5) エンドツーエンドのパイプラインを動かしてみる
make -C tests/e2e smoke
```

> 各 `subsystems/*/` や `tests/` 配下にも `Makefile` があります。
> `make` のターゲット、`tests/` の中身、`db/migration/` のスキーマは、**仕様を読み解く一次情報** です。
> Copilot に「この Makefile は何をする？」「このテストは何を保証している？」と聞いてみましょう。

## 📦 このリポジトリについて

- 実装言語: **COBOL**（GnuCOBOL でビルド、組み込み SQL は OCESQL 経由）
- データストア: **PostgreSQL** / メッセージング: **RabbitMQ**
- 形態: 複数プログラムで構成される **バッチ指向の業務アプリケーション**
- **ドキュメント・コメントは意図的に除去済み**（Quest の解析対象とするため）。残っているのは
  実行に必要なコードとビルド/テスト資材のみです。

それ以外が「何のアプリで、どう動き、どう作り変えるべきか」は —— **Copilot と一緒に解き明かしてください。**

**Happy modernizing. 🚀**
